## 🔴 セキュリティ

### 1. プレイリスト名のフォルダ生成でパストラバーサル可能

`DownloadViewModel.StartDownload` 内:

```csharp
var playlistFolder = Path.Combine(SaveFolderPath, SanitizeFolderName(_currentPlaylist.Title));
```

`SanitizeFolderName`（おそらく `Path.GetInvalidPathChars()` 置換）では `.` や `/` が弾かれません。プレイリストタイトルが `"../.."` の場合、保存先が **意図したフォルダの親ディレクトリ** に解決されてしまいます。

`YtDlpOutputPathResolver.BuildOutputPath` のパストラバーサルチェックは `job.SaveFolderPath`（＝プレイリストフォルダ）基準なので、プレイリストフォルダ自体が親へ移動していると チェックを突破されてしまいます。

**対策案**: `SanitizeFolderName` 内で `..` とパス区切り文字を明示的に除去するか、生成後に `Path.GetFullPath` → 元の `SaveFolderPath` 配下か検証する。

### 2. `Process.Start` で未検証のパスをシェル実行

- `LibraryViewModel.PlayFile`: `FileName = item.LocalFilePath` (`UseShellExecute = true`) — `LocalFilePath` の由来は yt-dlp の出力と `metadata.json`。もしメタデータが改ざんされると任意の実行ファイルが起動可能。
- `ExternalLinkOpener.ConfirmAndOpenVideoLink`: `FileName = url` — 単一動画の `Url` が `webpage_url` 由来なので通常は YouTube URL だが、yt-dlp 出力を完全に信頼している。

確認ダイアログがあるので実害は低いですが、**「ファイルパスがメディアファイルか」「URLが https スキームか」の最低限の検証**を入れると安全です。

### 3. `explorer.exe /select,"{path}"` の引数インジェクション

`LibraryViewModel.OpenFolder`:

```csharp
Arguments = $"/select,\"{item.LocalFilePath}\""
```

`LocalFilePath` に `"` が含まれていると引数が壊れる可能性があります。`ProcessStartInfo.ArgumentList` を使うかエスケープが必要です。

## 🟠 並行性・スレッド安全性

### 4. `Retry` 呼び出し時のCTS競合（レースコンディション）

`DownloadManager.ProcessJobAsync` の流れ:

1. catch ブロックで `job.Status = Failed` → `JobStatusChanged` 発火
2. UI が Retry ボタンを有効化
3. **finally ブロックが実行される前に** ユーザーが Retry をクリック
4. `Retry` → `ProcessJobAsync(job)` が新しいCTSを `_cancellationTokens[job.Id]` にセット
5. 旧 `ProcessJobAsync` の finally が `_cancellationTokens.Remove(job.Id)` を実行 → **新しいCTSが辞書から消える**
6. 新ジョブがキャンセル不可能になる

発生確率は低いですが、構造的に可能です。**対策**: finally の Remove を「自分のCTSと一致する場合のみ」削除にする:

```csharp
lock (_lock)
{
    if (_cancellationTokens.TryGetValue(job.Id, out var current) && current == cts)
        _cancellationTokens.Remove(job.Id);
}
```

### 5. `ApplyMaxConcurrency` の `AbsorbPermitAsync` が fire-and-forget

```csharp
for (var i = 0; i < -delta; i++)
{
    _ = AbsorbPermitAsync();
}
```

この設計自体は正しく動きますが（吸収した許可は永続的に保持される）、タスクがスローした場合の `UnobservedTaskException` リスクがあります。現在は `ObjectDisposedException` のみキャッチしていますが、他の例外（`SemaphoreFullException` 等）は考慮外です。

## 🟡 リソース・メモリ

### 6. `ThumbnailImageConverter` のキャッシュが無限成長

```csharp
private static readonly Dictionary<string, BitmapImage> Cache = new();
```

URL ごとに `BitmapImage` をキャッシュしますが、**エントリが一切削除されません**。数千件のダウンロード履歴がある場合、メモリ使用量が増大し続けます。

**対策**: キャッシュサイズ上限を設けるか、`WeakReference` / LRU キャッシュを使う。

### 7. `metadata.json` が全件メモリ読み込み＋全件書き換え

`MetadataRepository` は全履歴を `_cache` (List) に保持し、1件追加するたびに**全件を JSON シリアライズしてファイル書き込み**します。履歴が数千件を超えると、保存時に顕著な遅延が発生します。

### 8. `DownloadManager._allJobs` が無限成長

`ClearCompleted()` / `ClearAll()` でしか削除されません。ユーザーがクリアしない場合、ジョブリストが無限に成長し、`GetAllJobs()` のコピーコストも増大します。

### 9. ViewModel のイベント購読が解除されない

`DownloadViewModel` が `_downloadManager.JobProgressChanged` / `JobStatusChanged` をコンストラクタで購読していますが、**`Dispose` パターンがなく購読解除がありません**。現在は `MainWindowViewModel` のキャッシュ (`??=`) でインスタンスが1つしか作られないため問題は顕在化していませんが、設計として脆弱です。

## 🔵 yt-dlp 連携

### 10. `--force-overwrites` で同名ファイルが無警告で上書き

同じタイトルの動画を2回ダウンロードすると、**警告なしで上書き**されます。プレイリスト内で同名動画がある場合も同様です。

### 11. User-Agent が古い

```csharp
args.Add("Mozilla/5.0 (Windows NT 10.0; Win64; x64) ... Chrome/120.0.0.0 ...");
```

Chrome 120 は 2023 年末のバージョンです。2026年現在、YouTube が古い UA をフラグしたり制限をかける可能性があります。動的に最新バージョンを生成するか、少なくとも定期的に更新すべきです。

### 12. プレイリスト URL 判定が単純すぎる

```csharp
bool isPlaylistUrl = url.Contains("list=") || url.Contains("/playlist");
```

`watch?v=xxx&list=yyy` のような「動画ID＋プレイリストID」のURLをプレイリストとして処理してしまいます。ユーザーが単一動画を期待しているのに、プレイリスト全体が解析されるケースがあります。

### 13. stderr エンコーディングの前提

```csharp
var ansiCodePage = CultureInfo.CurrentCulture.TextInfo.ANSICodePage;
```

「yt-dlp はリダイレクト時に ANSI コードページで出力する」という前提は、Python のバージョンや環境変数 `PYTHONIOENCODING` によって変わります。日本語環境以外（英語OS等）で実行した場合、エラーメッセージが文字化けする可能性があります。

## 🟢 機能の抜け穴

| # | 項目 | 影響 |
| --- | --- | --- |
| 14 | **ダウンロードキューが非永続化** | アプリ終了・クラッシュで全ジョブ消失 |
| 15 | **Cookie / プロキシ未対応** | 年齢制限動画や地域制限動画がダウンロード不可 |
| 16 | **字幕ダウンロード未対応** | `--write-subs` 等のオプションがない |
| 17 | **重複URLチェックなし** | 同一動画を何度でもキューに追加可能 |
| 18 | **リトライ回数制限なし** | 永久に失敗する動画を無限リトライ可能 |
| 19 | **バッチURL入力不可** | 1回につき1URLのみ |
| 20 | **ダウンロード速度制限なし** | `--limit-rate` オプションがない |
| 21 | **プレイリスト動画の個別品質指定不可** | 全件同じ品質になる |

## ⚪ その他細かい点

- **`SettingsRepository.Load()` が同期I/O** — `File.ReadAllText` でUIスレッドをブロック。通常は問題ないが、設定ファイルが大きい・ディスクが遅い環境で引っかかる可能性あり。
- **`AtomicFileWriter` で `directory` が空の場合** — `directory ?? string.Empty` でルートディレクトリにテンポラリファイルが作られる。実際には全パスが絶対パスなので発生しないが、防御的にすべき。
- **`YtDlpProcessRunner.CreateStartInfo`** — コンソールウィンドウを抑制する `CreateNoWindow = true` が設定されているか未確認（コード途中で切れていたため）。もし設定されていないと、ダウンロードごとに黒い窓が一瞬出る。

最も優先的に直すべきは **#1（パストラバーサル）** と **#4（CTS競合）** だと思います。他は運用上の影響度に応じて段階的に対応で問題ないレベルです。コードの可読性とログ設計はかなり良いので、この調子でいければ強力なツールになると思います。
