サービス層、モデル、ViewModel、App/Infrastructure、View まわりを確認し、コードベース全体の構造を踏まえて評価しています。

## 【高】1. ダウンロードのキャンセルが yt-dlp プロセスを停止していない

**1. 改善すべき点**
[YtDlpClient.cs:485-540](YouTubeDownloader/Services/YtDlpClient.cs#L485) の `RunDownloadProcessAsync` では、`CancellationToken` を `Task.Run(..., token)` と `process.WaitForExitAsync(token)` に渡しているだけで、実際の `yt-dlp.exe` プロセスを `Kill` していません。さらに処理順序が `await Task.WhenAll(outputTask, errorTask)` → `await process.WaitForExitAsync(token)` となっており、stdout/stderr の読み取りループ（`ReadLineAsync()` にトークン未指定）はプロセスが終了して EOF に達するまで完了しません。

結果として、実行中ジョブの「キャンセル」を押しても **プロセスは終わらず、ダウンロードは最後まで走り切ってから** 例外で `Canceled` 扱いになります（実体は完走後にキャンセル表示）。

**2. 優先度**：高

**3. 理由**
「キャンセル」がユーザー期待どおりに即時停止しない、という中核機能の不具合です。大きなファイルや低速回線では帯域・ディスクを浪費し続けます。また `Cancel` を多数発行しても並列数（セマフォ）が解放されないため、後続ジョブが詰まります。

**4. 具体的な対応案**
キャンセル要求を受けた時点で、起動済みの `yt-dlp` プロセスとその子プロセスを明示的に終了させる仕組みを登録します。標準出力と標準エラーの読み取り処理は、プロセス終了時に確実に抜けられる形にして、終了後にキャンセル状態をジョブ側へ正しく伝える流れにします。

**5. 注意点**
kill 後は `WaitForExitAsync` が `OperationCanceledException` ではなく正常復帰し得るため、`cancellationToken.ThrowIfCancellationRequested()` を最後に挟むか、kill 経由の終了コード非0を `ProcessJobAsync` の `OperationCanceledException` 経路に確実に乗せる必要があります。`Kill(entireProcessTree: true)` は ffmpeg 子プロセスも巻き込むため意図どおりですが、共有 ffmpeg を別用途で使っている等の特殊ケースがないか確認してください。

## 【高】2. Pending ジョブのキャンセルがセマフォ解放後に無効化される

**1. 改善すべき点**
[DownloadManager.cs:80-83](YouTubeDownloader/Services/DownloadManager.cs#L80) で `CancellationTokenSource` を生成・登録するのは「セマフォ取得後」です。一方 [Cancel](YouTubeDownloader/Services/DownloadManager.cs#L134) の Pending 分岐は、CTS がまだ無いため `job.Status = Canceled` を立てるだけです。

しかし該当ジョブの `ProcessJobAsync` は既に `await _semaphore.WaitAsync()` で待機中で、セマフォが空くと **キャンセル済みであることを確認せずに** `job.Status = Running` を上書きしてダウンロードを開始します。`CancelAll` も同じ穴があります。

**2. 優先度**：高

**3. 理由**
「待機中ジョブをキャンセルしたのに後で勝手に動き出す」という到達不能であるべき状態への遷移で、再現性のある論理矛盾です。

**4. 具体的な対応案**
CTS を「セマフォ取得前」に生成・登録し、取得後にトークンを確認します。これで Pending/Running を統一的に扱え、`Cancel` の Pending 専用分岐も不要になります。
具体的には、ジョブ処理の開始直後にキャンセル用の管理情報を作成して登録し、そのトークンを使ってセマフォの待機自体も中断できるようにします。セマフォ取得後は、ダウンロード開始前に必ずキャンセル済みか確認し、待機中に取り消されたジョブを `Running` に戻さないようにします。

**5. 注意点**
セマフォ取得前にキャンセルされた場合、`WaitAsync(cts.Token)` を使えば取得待ち自体も打ち切れますが、その際はセマフォを `Release` してはいけない点に注意（取得していないため）。`Cancel` 側の Pending 専用分岐を削除する場合、イベント発火タイミングが変わるので UI 表示の差分を確認してください。

## 【高】3. `StartDownload` の `Directory.CreateDirectory` が未保護でクラッシュし得る

**1. 改善すべき点**
[DownloadViewModel.cs:219](YouTubeDownloader/ViewModels/DownloadViewModel.cs#L219) の `Directory.CreateDirectory(SaveFolderPath)` は UI スレッド上で try/catch なしに実行されます。保存先がアクセス不可、ドライブ未接続、不正パス（例: 存在しないネットワークドライブ）の場合に `UnauthorizedAccessException` / `IOException` 等が送出され、未処理例外でアプリが落ちます。

**2. 優先度**：高（クラッシュ＝データ消失や利用不能に直結）

**3. 理由**
ユーザーが任意フォルダを指定でき、外部ストレージや権限の問題は日常的に発生します。解析・ダウンロード系は概ね try/catch されているのに、ここだけ無防備です。

**4. 具体的な対応案**
保存先フォルダの作成処理を例外処理で保護し、失敗した場合はエラーメッセージを表示してダウンロード開始処理を中断します。アプリ全体を落とすのではなく、ユーザーに保存先の権限やパスを見直してもらえる状態にします。

**5. 注意点**
影響は局所的で互換性問題はありません。プレイリスト用の `playlistFolder` は `YtDlpClient.DownloadAsync` 側の try/catch で拾われますが、ここでの早期検証を入れておくと UX が向上します。

## 【高】4. メタデータ／設定ファイルが非アトミック書き込みで破損リスク

**1. 改善すべき点**
[MetadataRepository.cs:65-69](YouTubeDownloader/Services/MetadataRepository.cs#L65) と [SettingsRepository.cs:53-57](YouTubeDownloader/Services/SettingsRepository.cs#L53) は `File.WriteAllTextAsync` で直接上書きします。書き込み途中でクラッシュ／電源断が起きると JSON が壊れ、`LoadIfNeededAsync` の `catch` で **履歴が全件消えます**（空リストにフォールバック）。

**2. 優先度**：高（ダウンロード履歴という蓄積データの全損リスク）

**3. 理由**
`SaveVideoMetadataAsync` はダウンロード完了のたびに全件を毎回シリアライズ・上書きするため、書き込み頻度が高く破損ウィンドウが相対的に大きいです。

**4. 具体的な対応案**
一時ファイルに書いてから `File.Replace`/`Move` で原子的に差し替えます。
保存時は、まず一時ファイルへ JSON を最後まで書き出し、書き込みが成功した後で元ファイルと差し替える流れにします。これにより、書き込み途中のクラッシュや電源断が起きても、既存ファイルを壊す可能性を下げられます。

**5. 注意点**
`File.Move(overwrite:true)` は .NET Core 3.0+ で利用可。ウイルス対策ソフトのロックで稀に失敗するため、失敗時のリトライ/握り潰し方針は要検討。バックアップ世代（`.bak`）まで作るかは要件次第。

## 【高】5. `MaxConcurrentDownloads` 設定が実装に接続されていない（仕様乖離・デッド設定）

**1. 改善すべき点**
[AppSettings.cs:44](YouTubeDownloader/Models/AppSettings.cs#L44) に `MaxConcurrentDownloads` がありますが、`DownloadManager` は [const int MaxConcurrency = 2](YouTubeDownloader/Services/DownloadManager.cs#L42) を使っており設定値を一切参照しません。さらに `SettingsViewModel`／設定画面にこの項目の UI もありません。「設定モデルにあるが効かない」死んだ設定です。

**2. 優先度**：高（設定とコードの明確な乖離）

**3. 理由**
将来この値を信じて挙動を組むと不整合の温床になります。放置するなら少なくとも UI/モデルから削除すべきです。

**4. 具体的な対応案**
本当に可変にするなら、`DownloadManager` をコンストラクタで設定から初期化します。
具体的には、`DownloadManager` の生成時に設定値を読み込み、許容範囲に丸めたうえでセマフォの並列数として使います。可変にしない方針であれば、設定モデルからこの項目を削除して、実装と仕様を一致させます。

実行中の動的変更まで対応するかは要判断（新しい `SemaphoreSlim` への差し替えは慎重に）。可変にしないなら設定/プロパティを削除。

**5. 注意点**
`DownloadManager` は Singleton なので起動時の値で固定になります。「設定変更を即時反映」を期待されると別問題になるため、当面は「再起動後に反映」と割り切るのが安全。**この設定を最終的に「使う」のか「消す」のか、意図の確認が必要**です。

## 【中】6. `duration` が小数の場合に `GetInt32()` で解析全体が失敗する

**1. 改善すべき点**
[YtDlpClient.cs:248-262](YouTubeDownloader/Services/YtDlpClient.cs#L248) の `GetIntProperty` は数値に対し `prop.GetInt32()` を呼びます。yt-dlp の JSON は `duration` を `184.0` のような小数で出すことがあり、その場合 `GetInt32()` は例外を投げます。これが `ParseVideoMetadata` の `catch` で握られて `null` → 「動画情報を取得できませんでした」、プレイリストでは [ParsePlaylistFromJson](YouTubeDownloader/Services/YtDlpClient.cs#L195) の外側 catch で**プレイリスト全体の解析失敗**になります。

**2. 優先度**：中（特定動画で解析が突然失敗、原因が分かりにくい）

**3. 理由**
duration の整数/小数は動画によって揺れるため、再現条件が分かりづらいバグになりがちです。

**4. 具体的な対応案**
数値を取得するときは、まず整数として読めるかを試し、整数で読めない場合は小数として読み取って秒数へ変換します。整数前提で例外を出すのではなく、yt-dlp 側の JSON 表現の揺れを吸収する形にします。

**5. 注意点**
影響範囲は `duration` 取得のみ。表示用の概算秒数なので `(int)` 切り捨てで問題ありません。**実際に小数が来るかは yt-dlp のバージョン依存のため、再現確認ができると確実**です。

## 【中】7. 403 リトライの分岐に到達不能コードがある（コメントで自認済み）

**1. 改善すべき点**
[YtDlpClient.cs:368-392](YouTubeDownloader/Services/YtDlpClient.cs#L368) で、最初の `if` が `stderr.Contains("403")` という非常に広い条件のため、後続の `else if (IsHttpForbidden(stderr))` は**常に到達しません**（コード内 NOTE でも明記）。本来「高画質形式が 403 で拒否されたら 360p へ自動低下せず停止して警告」する意図だった分岐が死んでいます。

**2. 優先度**：中

**3. 理由**
コメント・関数名（`IsHttpForbidden`）と実挙動が乖離しており、保守者の誤読を招きます。意図した「高画質 403 は停止して警告」が機能していない可能性があります。

**4. 具体的な対応案**
方針を確定して片方に寄せます。
- 「全 403 を android で再試行する」が現方針なら、`IsHttpForbidden` 分岐と関連メッセージを削除。
- 「高画質 403 は停止」を活かすなら、再試行とは別条件（例: フォーマット未取得 `Requested format is not available`）で判定し直し、`403` の広すぎる部分文字列一致を `HTTP Error 403` に絞る。

**5. 注意点**
**どちらの挙動が現行の正なのか仕様確認が必要**です。403 時の「android 再試行」と「停止警告」は相反するため、ユーザー体験の優先順位を決める必要があります。

## 【中】8. ダウンロード完了後のファイル特定が `files[0]` で曖昧

**1. 改善すべき点**
[UpdateLocalFilePath](YouTubeDownloader/Services/YtDlpClient.cs#L545) は `Directory.GetFiles(folder, "{base}.*")` の先頭要素を `LocalFilePath` にします。同名ベースで複数ファイルが残る場合（`.part` 残骸、サムネ `.jpg`、中間 `.webm` など）、本来の成果物でないファイルを指す可能性があります。`files[0]` は列挙順に依存し決定的でもありません。

**2. 優先度**：中

**3. 理由**
ライブラリの「再生」「フォルダで表示」が誤ったファイルを開く原因になります。

**4. 具体的な対応案**
想定拡張子を優先し、無ければ最終更新時刻が最新のものを選ぶ。
ダウンロード後に同じベース名のファイルが複数見つかった場合は、まずユーザーが選択した形式と一致する拡張子を優先します。それでも候補が複数ある場合は、最終更新時刻が最も新しいファイルを成果物として扱うようにします。

**5. 注意点**
`job.Format` は `NormalizeFormat` 前の生値なので、音声時は実拡張子（mp3/m4a/wav）と一致するよう正規化後の値を渡すと精度が上がります。タイトルに `[` 等を含む場合の検索パターンは .NET では `*`/`?` のみワイルドカードなので実害は小さいです。

## 【中】9. プロセス引数を単一文字列で組み立てており、引用符・特殊文字に脆い

**1. 改善すべき点**
[BuildDownloadArguments](YouTubeDownloader/Services/YtDlpClient.cs#L401) などは `-o "{outputPath}"` のように手動でクォートした 1 本の文字列を `ProcessStartInfo.Arguments` に渡します。保存先パスやタイトルに `"` が含まれると引数解釈が壊れます（YouTube URL では稀ですが、ユーザー指定パスでは起こり得る）。

**2. 優先度**：中（保守性・堅牢性）

**3. 理由**
将来の引数追加でクォート漏れ・スペース起因のバグが発生しやすく、エスケープ責務が散在します。

**4. 具体的な対応案**
`ProcessStartInfo.ArgumentList` に移行すると .NET が自動エスケープします。
引数全体を 1 本の文字列として組み立てるのではなく、オプション名、出力先パス、フォーマット指定、URL などを個別の引数として追加する方式へ移行します。これにより、スペースや引用符を含むパスでも .NET 側にエスケープを任せられます。

**5. 注意点**
`--extractor-args "youtube:lang=ja;..."` のように `;` を含む値も `ArgumentList` ではクォート不要でそのまま渡せます。`BuildAndroidRetryArguments` が「文字列置換」前提なので、`ArgumentList`（List 操作）ベースへ再設計が必要です。**やや広範な変更**になるため段階的導入を推奨。

## 【中】10. `DefaultAudioFormat` 設定が初期選択に反映されない

**1. 改善すべき点**
設定の `DefaultAudioFormat` は保存・読込されますが、[DownloadViewModel](YouTubeDownloader/ViewModels/DownloadViewModel.cs#L42) の初期化は `DefaultVideoFormat` のみ使用。`IsAudioOnly` 切替時は [OnIsAudioOnlyChanged](YouTubeDownloader/ViewModels/DownloadViewModel.cs#L115) が `"mp3"`/`"mp4"` をハードコードします。結果、`DefaultAudioFormat` や `DefaultVideoFormat="mkv"` 等のユーザー設定が音声/動画切替時に無視されます。

**5. 注意点**
不正な保存値（リストにないフォーマット）への防御として、`CurrentFormats` に含まれなければ先頭にフォールバックする処理を入れると安全です。

## 【中】11. アプリ終了時に `ServiceProvider`/`DownloadManager` が破棄されず、子プロセスが孤児化

**1. 改善すべき点**
[App.xaml.cs](YouTubeDownloader/App.xaml.cs#L50) は `ServiceProvider` を破棄せず、`DownloadManager.Dispose`（[IDisposable 実装あり](YouTubeDownloader/Services/DownloadManager.cs#L212)）も呼ばれません。加えて指摘1の通り Dispose 内の `cts.Cancel()` ではプロセスを kill しないため、ダウンロード中にウィンドウを閉じると `yt-dlp.exe`/`ffmpeg.exe` が残ります。

**2. 優先度**：中

**3. 理由**
孤児プロセスがバックグラウンドで帯域・CPU・ディスクを使い続け、次回起動時の更新やファイルロックの原因になります。

**4. 具体的な対応案**
`OnExit` で破棄。指摘1の kill 登録と組み合わせれば、Cancel→プロセス終了が機能します。
アプリ終了時の処理で DI コンテナを明示的に破棄し、そこから `DownloadManager` の破棄処理が呼ばれるようにします。指摘1のプロセス終了対応と組み合わせることで、アプリ終了時にも実行中の子プロセスを残さない流れにできます。

**5. 注意点**
`Dispose` のキャンセルが実プロセス終了に結びつくのは指摘1の修正が前提です。終了時に確認ダイアログ（実行中ジョブあり）を出すかは UX 方針次第。

## 【中】12. `watch?v=...&list=...` の単一動画 URL がプレイリスト全体として解析される

**1. 改善すべき点**
[AnalyzeUrlAsync](YouTubeDownloader/Services/YtDlpClient.cs#L101) は `url.Contains("list=")` でプレイリスト判定します。YouTube の通常の視聴 URL はミックス/再生リスト由来の `&list=` を含むことが非常に多く、その場合「1 本の動画を入れたのにプレイリスト一覧が出る」挙動になります。

**2. 優先度**：中（仕様確認が必要）

**3. 理由**
意図せず巨大プレイリスト解析が走り、待ち時間増・誤操作の原因になります。一方で「リスト付き URL は一覧で扱う」が仕様なら現状で正です。

**4. 具体的な対応案**
`/playlist` パスや純粋なリスト URL のみプレイリスト扱いにし、`watch?v=` を伴う場合は単一動画優先にする、もしくは UI で「単一/一覧」を選ばせる。
判定ルールとしては、明確なプレイリストページや純粋なリスト URL のみをプレイリストとして扱い、通常の視聴 URL にリスト情報が付いている場合は単一動画を優先する形が考えられます。既存の使い方を尊重するなら、解析前に UI で「単一動画として扱う」か「プレイリストとして扱う」かを選べるようにします。

**5. 注意点**
**現行が意図的かどうか確認が必要**です。挙動を変えると「リスト付き視聴 URL から一覧を出す」既存ワークフローに影響します。

## 【中】13. `Retry` が `_lock` 保持中にイベント発火・ジョブ開始を行う（`Enqueue` と非対称）

**1. 改善すべき点**
[Enqueue](YouTubeDownloader/Services/DownloadManager.cs#L63) は `lock` の外で `_ = ProcessJobAsync(job)` を呼びますが、[Retry](YouTubeDownloader/Services/DownloadManager.cs#L177) は `lock(_lock)` の内側で `JobStatusChanged` 発火と `ProcessJobAsync` 起動を行います。セマフォに空きがある場合、`ProcessJobAsync` の同期部分（Running への遷移とイベント発火）がロック保持中に同一スレッドで走り得ます。イベントハンドラがマネージャへ再入すると、再入ロック自体は許容でもデッドロックや想定外の処理順を招きやすい構造です。

**2. 優先度**：中

**3. 理由**
ロック中の外部コールバック発火は WPF/イベント駆動でのデッドロック・再入バグの典型パターンで、`Enqueue` と挙動が非対称な点も保守性を下げます。

**4. 具体的な対応案**
`ProcessJobAsync` の起動をロックの外へ。
ロック中は対象ジョブの取得と状態更新だけを行い、イベント通知とジョブ再開処理はロックを抜けた後に実行します。これにより、イベントハンドラからの再入や UI 側の処理がロック保持中に走るリスクを避けられます。

**5. 注意点**
イベント発火順がわずかに変わります。UI 側は `UpdateFromJob` で冪等に更新するため実害は小さい見込みですが、回帰確認を推奨。

## 【低】14. 設定ファイルの読込がキャッシュされず、操作ごとに毎回ディスク I/O

**1. 改善すべき点**
[SettingsRepository.Load](YouTubeDownloader/Services/SettingsRepository.cs#L36) はキャッシュを持たず、`YtDlpClient` 内だけでも `GetYtDlpPath`／`AnalyzeUrlAsync`／`DownloadAsync`／`EnsureYtDlpUpdatedAsync` で各々 `Load()` を呼び、1 ダウンロードあたり複数回 JSON を読み直します。

**2. 優先度**：低（実害は軽微だが無駄）

**3. 理由**
小さなファイルとはいえ毎回パース。並列ダウンロード時に頻度が増えます。メモリにキャッシュし `SaveAsync` で無効化すれば十分です。

**4. 具体的な対応案**
`SettingsRepository` に `_cached` を持たせ、`Load` で再利用、`SaveAsync` で更新（Singleton 登録済みなので寿命も適合）。あるいは `DownloadAsync` 冒頭で一度だけ `Load()` し、内部メソッドへ渡す。

**5. 注意点**
「設定画面で保存したら次のダウンロードに反映」が必要なので、`SaveAsync` 後にキャッシュ更新を忘れない設計にすること。

## 【低】15. ライブラリの `Process.Start`（再生/フォルダ表示）が try/catch 未保護

**1. 改善すべき点**
[PlayFile](YouTubeDownloader/ViewModels/LibraryViewModel.cs#L87) と [OpenFolder](YouTubeDownloader/ViewModels/LibraryViewModel.cs#L70) の `Process.Start` は、関連付け無し・権限不足などで `Win32Exception` を投げ得ます（`File.Exists` 後でも起動失敗はあり得る）。`ExternalLinkOpener` は try/catch 済みですが、こちらは未保護です。

**2. 優先度**：低

**3. 理由**
拡張子に関連付けアプリが無い環境などでアプリが落ちます。

**4. 具体的な対応案**
`ExternalLinkOpener` と同様に try/catch で `MessageBox` 表示。

**5. 注意点**
影響は局所的、互換性問題なし。

## 【低】16. 細かな保守性・堅牢性（まとめて）

**1. 改善すべき点**（複数）
- [DownloadManager.cs:37](YouTubeDownloader/Services/DownloadManager.cs#L37) の `_pendingQueue` は宣言only で未使用（デッドコード）。
- [ParseProgressLine](YouTubeDownloader/Services/YtDlpClient.cs#L794) の `if (start < 0)` は `LastIndexOf(...) + 1` が最小 0 のため**到達不能**。
- `VideoFormats`/`AudioFormats`/`Qualities` 配列が [DownloadViewModel](YouTubeDownloader/ViewModels/DownloadViewModel.cs#L109) と [SettingsViewModel](YouTubeDownloader/ViewModels/SettingsViewModel.cs#L84) に二重定義（定義の一元化余地）。
- `_cachedYtDlpPath`/`_cachedFfmpegPath`（[YtDlpClient.cs:42-43](YouTubeDownloader/Services/YtDlpClient.cs#L42)）は static 可変フィールドを並列ダウンロードから無同期アクセス（代入は冪等なので実害は小だがデータ競合）。
- [SanitizeFilename](YouTubeDownloader/Services/YtDlpClient.cs#L774) は Windows 予約名（CON, NUL, COM1 等）・末尾のドット/空白・パス長上限・空タイトル（→ `.%(ext)s` の隠しファイル化）を未考慮。
- 進捗通知が [Dispatcher.Invoke](YouTubeDownloader/ViewModels/DownloadViewModel.cs#L372)（同期）で毎行 UI スレッドへマーシャル。高速進捗で UI スレッドが詰まり得るので `BeginInvoke`/`InvokeAsync` が無難。

**2. 優先度**：低

**3. 理由**
個々の実害は小さいものの、デッドコード・到達不能分岐・重複定義は誤読の温床、予約名/長さ未対応は特定タイトルでの保存失敗を招きます。

**4. 具体的な対応案**
デッドコード/到達不能分岐は削除。配列定義は共有定数クラスへ集約。`SanitizeFilename` に予約名チェックと空文字フォールバック（例: 動画 ID 採用）、長さ制限を追加。進捗マーシャルは `BeginInvoke` 化。

**5. 注意点**
`SanitizeFilename` 変更は生成ファイル名が変わるため、既存の「同名上書き」挙動（`--force-overwrites`）との整合を確認。配列一元化は XAML バインディング名の変更に注意。
