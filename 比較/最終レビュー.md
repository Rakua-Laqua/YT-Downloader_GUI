# 最終コードレビュー（統合版）

前回（`評価内容.md`）でカバーされている指摘、および直近の差分で **前回 指摘1（キャンセルが yt-dlp プロセスを止めない）・指摘2（Pending キャンセルの無効化）・指摘3（`Directory.CreateDirectory` 未保護）が修正済み** であることをコード上で確認しました（[YtDlpClient.cs:505](YouTubeDownloader/Services/YtDlpClient.cs#L505) の kill 登録、`ProcessJobAsync` のセマフォ取得前 CTS 生成＋取得後の `ThrowIfCancellationRequested`、`StartDownload` の try/catch）。

そのうえで、**修正済み項目を除いてなお残る重要項目** と、**前回挙がっていない新しい観点** を優先度順にまとめます。単体の最終レビューとして読めるよう、前回指摘済みでも未解決かつ影響が大きいものは再掲します。

> 補足：中核機能（キャンセル）の不具合は確かに解消済みですが、**実ファイルの損失や永続データ破損につながり得る高優先度の問題はまだ残っています**。「致命的な問題はもう無い」とは言い切れないため、まずそこから記載します。

## 【高】1. `--force-overwrites` ＋ ファイル名衝突で、既存ファイル／別動画を確認なく上書きし得る

**1. 改善すべき点**
[BuildDownloadArguments](YouTubeDownloader/Services/YtDlpClient.cs#L401) は常に `--force-overwrites` を付与します（[YtDlpClient.cs:408](YouTubeDownloader/Services/YtDlpClient.cs#L408)）。出力名は [BuildFilenameTemplate](YouTubeDownloader/Services/YtDlpClient.cs#L761) が生成しますが、

- (a) 既定テンプレート `{title}` 単独だと、**同名タイトルの動画を含むプレイリスト** で出力名が衝突します。
- (b) ユーザーが `{index}` だけ等の縮退テンプレートを設定し、単一動画で `PlaylistIndex` が無い場合、`{index}` が空に置換され（[L775](YouTubeDownloader/Services/YtDlpClient.cs#L775)）、ベース名が空 → 拡張子のみの `.%(ext)s`（[L783](YouTubeDownloader/Services/YtDlpClient.cs#L783)）になり得ます。

いずれも `--force-overwrites` と組み合わさると、同名出力が **既存ファイルや先行ダウンロード分を確認なく上書き** します。

**2. 優先度**：高

**3. 理由**
履歴の不整合に留まらず、**実ファイルの消失** に直結します。しかも (a) は既定テンプレートのままでも（同名タイトルを含むプレイリストで）普通に起こり得ます。

**4. 具体的な対応案**
- `BuildFilenameTemplate` で、最終ベース名が空または拡張子のみになる場合は、必ず `Id`（またはタイトル）を入れる防御を追加する。
- 一意性の低いテンプレート（複数動画に対する `{title}` 単独、`{index}` 単独 等）は **設定保存時に検証** し、警告するか `{title}_{id}` 等へフォールバックする。
- そもそも `--force-overwrites` が必須か再検討する（少なくとも「同名が既存なら衝突回避リネーム」を選べると安全）。

**5. 注意点**
既存ユーザーが固定名保存を意図している可能性があるため、いきなり強制変更せず **警告から始める** のが無難です。前回 指摘16 の `SanitizeFilename` 空文字フォールバック（空タイトル → 隠しファイル化）と整合させてください。

## 【高】2. メタデータ／設定ファイルが非アトミック書き込みで、破損時に履歴や設定を失い得る

**1. 改善すべき点**
[MetadataRepository.cs:64-68](YouTubeDownloader/Services/MetadataRepository.cs#L64) と [SettingsRepository.cs:53-57](YouTubeDownloader/Services/SettingsRepository.cs#L53) は `File.WriteAllTextAsync` で JSON を直接上書きしています。書き込み中にクラッシュ・電源断・ウイルス対策ソフトの一時ロック等が起きると、`metadata.json` / `settings.json` が途中までの内容になり得ます。さらに `MetadataRepository.LoadIfNeededAsync` は読み込み失敗時に空リストへフォールバックするため（[L57-L60](YouTubeDownloader/Services/MetadataRepository.cs#L57)）、破損した履歴ファイルが **全件消えたように見える** 挙動になります。

**2. 優先度**：高

**3. 理由**
履歴と設定はアプリの永続データです。特に `SaveVideoMetadataAsync` はダウンロード完了ごとに全件を再シリアライズして上書きするため、書き込み頻度が高く、破損時の影響も大きいです。

**4. 具体的な対応案**
一時ファイルへ最後まで書き切ってから、`File.Replace` または `File.Move(overwrite: true)` で原子的に差し替えます。読み込み失敗時は即座に空リストへ確定せず、破損ファイルを `.bak` として退避し、ユーザーに復旧可能性を示すのが安全です。

**5. 注意点**
`MetadataRepository` は mutex により同一プロセス内の同時保存は保護できていますが、これは「同時書き込み競合」の対策であり、「書き込み途中のプロセス終了や I/O 失敗によるファイル破損」の対策ではありません。

## 【高】3. `MaxConcurrentDownloads` 設定が実装に接続されていない

**1. 改善すべき点**
[AppSettings.cs:44](YouTubeDownloader/Models/AppSettings.cs#L44) に `MaxConcurrentDownloads` がありますが、`DownloadManager` は [DownloadManager.cs:42](YouTubeDownloader/Services/DownloadManager.cs#L42) の `private const int MaxConcurrency = 2;` を使っており、設定値を参照していません。設定モデル上は可変に見えるのに、実際の同時ダウンロード数は常に 2 固定です。

**2. 優先度**：高

**3. 理由**
設定値が存在するのに実装へ接続されていない状態は、仕様とコードの明確な乖離です。将来この値を UI や設定ファイルで変更できるようにした場合、ユーザーの期待と実挙動が食い違います。

**4. 具体的な対応案**
本当に可変にするなら、`DownloadManager` の生成時に設定値を読み、許容範囲に丸めた上で `SemaphoreSlim` の初期値として使います。起動中の動的変更まで対応するかは別判断でよく、まずは「起動時に反映」だけでも設定と実装が一致します。可変にしない方針なら、`AppSettings` から `MaxConcurrentDownloads` を削除して死んだ設定をなくします。

**5. 注意点**
`DownloadManager` は Singleton のため、起動後に設定画面で変更した値を即反映するには追加設計が必要です。まず「設定を消す」か「次回起動後に反映する」かを決めると安全です。

## 【中】4. 進捗パースの `double.TryParse` がカルチャ非依存でなく、一部ロケールで進捗が壊れる

**1. 改善すべき点**
[YtDlpClient.cs:812](YouTubeDownloader/Services/YtDlpClient.cs#L812) の `if (double.TryParse(percentStr, out var percent))` はカルチャ未指定です。この overload は `CultureInfo.CurrentCulture` を使うため、**小数点にカンマを用いるロケール（ドイツ語・フランス語など）** では yt-dlp が出力する `"50.0"` の `.` が桁区切りと解釈され、`50.0 → 500` になったりパース失敗（0%）になります。同ファイルの `upload_date` 解析（[L216](YouTubeDownloader/Services/YtDlpClient.cs#L216)）は `CultureInfo.InvariantCulture` を使っており、ここだけ不整合です。

**2. 優先度**：中（日本語既定環境では影響なし。非日本語ロケール環境で顕在化）

**3. 理由**
yt-dlp の数値出力は常にドット小数なので、外部プロセス出力のパースは必ず InvariantCulture にすべきです。放置すると非日本語ロケール環境で「進捗バーが 0% のまま」「ステータスが `(500%)` 表示」といった再現条件の分かりにくい不具合になります。

**4. 具体的な対応案**
```csharp
if (double.TryParse(percentStr, NumberStyles.Float, CultureInfo.InvariantCulture, out var percent))
```
`ParseProgressLine` 内の数値パースを InvariantCulture 固定にします。

**5. 注意点**
日本語環境では現状でも `.` が小数点なので挙動は変わりません（既存動作維持）。影響は局所的で互換性問題なし。

## 【中】5. 自動更新／解析プロセスがキャンセル時に kill されない（download path との非対称）

**1. 改善すべき点**
前回 指摘1 の修正で `RunDownloadProcessAsync` には `cancellationToken.Register(() => process.Kill(...))`（[YtDlpClient.cs:505](YouTubeDownloader/Services/YtDlpClient.cs#L505)）が入りましたが、**`RunYtDlpUpdateAsync`（[L709](YouTubeDownloader/Services/YtDlpClient.cs#L709)）と `RunYtDlpAsync`（[L846](YouTubeDownloader/Services/YtDlpClient.cs#L846)）には kill 登録がありません**。特に `EnsureYtDlpUpdatedAsync` は `DownloadAsync` 内で **ジョブのトークン付き** で呼ばれる（[L349](YouTubeDownloader/Services/YtDlpClient.cs#L349)）ため、「初回ダウンロードの更新フェーズ中にキャンセル」を押すと、`ReadToEndAsync(token)`/`WaitForExitAsync(token)` が OCE を投げてジョブは Canceled になるものの、`yt-dlp -U` プロセス自体は走り続けます。

**2. 優先度**：中

**3. 理由**
前回 指摘1 で「ダウンロードは止まるが更新・解析は孤児化し得る」という抜けが残っています。`yt-dlp -U` は自己バイナリを置き換える処理なので、中断・孤児化はバイナリ破損や次回起動時のロックにつながる可能性があります。

**4. 具体的な対応案**
方針は2択で、**こちらは方針確認が必要** です。
- **更新は中断させない** のが安全（推奨）：`EnsureYtDlpUpdatedAsync` 内の更新呼び出しを `CancellationToken.None` で実行し、ダウンロードのキャンセルが自己更新を巻き込まないようにする。
- もしくは `RunDownloadProcessAsync` と同じ kill 登録を共通化して update/analyze にも適用する（ただし自己更新の途中 kill はバイナリ破損リスクがあるため、更新には不向き）。

**5. 注意点**
`-U` の途中 kill は逆効果になり得るため、安易に kill 登録を横展開せず「更新だけはキャンセル非対応にする」方が無難です。解析（`RunYtDlpAsync`）側は UI から token を渡していない（[DownloadViewModel.cs:143](YouTubeDownloader/ViewModels/DownloadViewModel.cs#L143)）ので現状キャンセル経路はなく、実害は更新パスが中心です。

## 【中】6. 解析用 `RunYtDlpAsync` が exit code と stderr を捨て、失敗理由が表に出ない

**1. 改善すべき点**
[RunYtDlpAsync](YouTubeDownloader/Services/YtDlpClient.cs#L846) は `outputTask.Result`（stdout）だけを返し（[L871](YouTubeDownloader/Services/YtDlpClient.cs#L871)）、exit code と stderr を破棄しています。`AnalyzeUrlAsync` は出力が空なら一律「動画情報を取得できませんでした。」（[L132](YouTubeDownloader/Services/YtDlpClient.cs#L132)）に丸めます。

**2. 優先度**：中

**3. 理由**
非公開動画・地域制限・削除済み・yt-dlp 側エラーなどが、すべて同じ汎用文言に丸められ、ユーザーも保守側も原因を追えなくなります。

**4. 具体的な対応案**
`RunYtDlpAsync` を `(ExitCode, StdOut, StdErr)` 返却に変え、`ExitCode != 0` の場合は stderr の末尾数行を `ErrorMessage` に含めます。stderr の有無ではなく **exit code を主判定** にします。

**5. 注意点**
stderr には警告も混ざるため判定は exit code を主に。ハング防止のため stderr を読み切る現行方式（[L863-866](YouTubeDownloader/Services/YtDlpClient.cs#L863)）はそのまま維持してください。

## 【中】7. プロセス引数を単一文字列で組み立てており、引用符・特殊文字に脆い

**1. 改善すべき点**
[BuildDownloadArguments](YouTubeDownloader/Services/YtDlpClient.cs#L389) や `AnalyzeUrlAsync` の呼び出しは、`-o "{outputPath}"` や `"\"{url}\""` のように手動でクォートした 1 本の文字列を `ProcessStartInfo.Arguments` に渡しています（[YtDlpClient.cs:470-474](YouTubeDownloader/Services/YtDlpClient.cs#L470)、[L824-L830](YouTubeDownloader/Services/YtDlpClient.cs#L824)）。保存先パス・テンプレート由来の文字列・URL に引用符や特殊文字が含まれると、引数解釈が壊れる可能性があります。

**2. 優先度**：中

**3. 理由**
現状の通常入力では顕在化しにくいものの、ユーザー指定パスや将来の引数追加時に壊れやすい構造です。手動クォートが散在すると、修正や機能追加のたびにエスケープ漏れを生みやすくなります。

**4. 具体的な対応案**
`ProcessStartInfo.ArgumentList` に移行し、オプション名・値・URL を個別の引数として追加します。`--extractor-args` のように `;` を含む値も、`ArgumentList` ならクォート不要でそのまま渡せます。既存の `BuildDownloadArguments` / `BuildAndroidRetryArguments` は、文字列置換ではなく `List<string>` を組み立てる形へ段階的に移行するのが安全です。

**5. 注意点**
一括置換すると影響範囲が広いため、まず download/analyze/update の共通プロセス起動ヘルパーを作り、既存の引数をテストしながら移行するのが無難です。`BuildAndroidRetryArguments` は現在「文字列置換」前提なので、同時に再設計が必要です。

## 【中】8. `DownloadViewModel`（Transient）が Singleton の `DownloadManager` イベントを購読し、解除しない

**1. 改善すべき点**
[DownloadViewModel.cs:37-38](YouTubeDownloader/ViewModels/DownloadViewModel.cs#L37) で `JobProgressChanged`/`JobStatusChanged` を購読していますが、`DownloadViewModel` は DI 上 `AddTransient`（[App.xaml.cs:34](YouTubeDownloader/App.xaml.cs#L34)）で、購読解除（`IDisposable`）もありません。現状は `MainWindowViewModel` が `_downloadViewModel ??= factory()`（[MainWindowViewModel.cs:53](YouTubeDownloader/ViewModels/MainWindowViewModel.cs#L53)）でキャッシュするため **実際には1個しか生成されず実害は出ていません**。ただ「Transient + イベント購読 + 非Dispose」の組み合わせは、将来ビューを作り直す設計に変えた瞬間にリーク＆二重発火（同じ進捗更新が複数 VM に飛ぶ）を招く地雷です。

**2. 優先度**：中（現状は潜在的、設計上の注意）

**3. 理由**
実態が Singleton なら DI 登録もそれに合わせる方が意図が明確で、誤った再利用を防げます。逆に Transient のままにするなら購読解除が必須です。

**4. 具体的な対応案**
いずれか。
- `DownloadViewModel`/`LibraryViewModel`/`SettingsViewModel` を `AddSingleton` に変更（`MainWindowViewModel` のキャッシュと整合）。
- もしくは `DownloadViewModel : IDisposable` にして `-=` で購読解除し、ビュー破棄時に Dispose する。

**5. 注意点**
Singleton 化すると各ビューの状態が常駐します（入力欄やキュー表示が保持され、むしろ望ましいことが多い）。`LibraryViewModel` だけは「開くたびに最新化」したい場合があるので、`NavigateToLibraryAsync` の `LoadAsync()` 呼び出しは現状維持で問題ありません。

## 【中】9. ライブラリがダウンロード完了を自動反映しない

**1. 改善すべき点**
`LibraryViewModel` は `DownloadManager` のイベントを購読していません。ダウンロード完了で `MetadataRepository` に履歴が追加されますが、**既にライブラリ画面を開いている状態では一覧が更新されません**（再ナビゲーション or 更新ボタンが必要）。

**2. 優先度**：中（機能の自然さに関わる）

**3. 理由**
「ダウンロード→ライブラリで確認」という基本動線で、ユーザーが「保存されていない？」と誤解しやすい挙動です。

**4. 具体的な対応案**
`LibraryViewModel` で `JobStatusChanged` を購読し、`Completed` 到達時に該当アイテムを `Items` に追加（または `RefreshAsync` を間引き実行）。Dispatcher マーシャルは `DownloadViewModel` 同様に。指摘8 で Singleton 化する場合は購読のライフサイクルも揃えてください。

**5. 注意点**
購読する場合は指摘8の購読解除問題が同じく付きまといます。常時 `RefreshAsync` は全件再読込なので、完了1件ごとの差分追加にする方が効率的です。

## 【低】10. `BoolToVis`（標準 `BooleanToVisibilityConverter`）の用途外利用で、空状態が表示されない

**1. 改善すべき点**
`BoolToVis` は標準の `BooleanToVisibilityConverter` です（[DownloadView.xaml:12](YouTubeDownloader/Views/DownloadView.xaml#L12) 他）。

- [DownloadView.xaml:322](YouTubeDownloader/Views/DownloadView.xaml#L322) と [LibraryView.xaml:177](YouTubeDownloader/Views/LibraryView.xaml#L177) は `int` の `Count` を渡し `ConverterParameter=Inverted` を期待していますが、標準コンバーターは bool 以外を false 扱いし `Inverted` も解釈しません。結果、**常に `Collapsed` となり「ダウンロードキューは空です」「ライブラリにアイテムがありません」が永久に出ません**。

**2. 優先度**：低（表示のみ）

**3. 理由**
空状態メッセージという地味だが実害のある UI 不具合に加え、出力ウィンドウにバインディングエラーが出続けます。

**4. 具体的な対応案**
- 空表示は ViewModel に `IsQueueEmpty` / `IsLibraryEmpty` を公開する（または `int→Visibility` 専用コンバーターを作る）。
- バッジ背景は `DataTrigger` で色を切り替える。

**5. 注意点**
`ObservableCollection.Count` の変化通知を確実に拾う必要があります（`IsQueueEmpty` を Count 変化時に `OnPropertyChanged`）。表示だけの変更で副作用は小さいです。

## 【低】12. `MaybeFlashWhenAllCompleted` が Canceled/Failed 混在時に永遠に発火しない

**1. 改善すべき点**
[DownloadViewModel.cs:408](YouTubeDownloader/ViewModels/DownloadViewModel.cs#L408) は `DownloadQueue.All(j => j.Status == Completed)` のため、キューに **1件でも Canceled/Failed があると、残り全部が完了してもタスクバー点滅通知が出ません**。

**2. 優先度**：低

**3. 理由**
「数十件中1件だけ失敗、残りは完了」でも完了通知が来ないのは、通知機能としては取りこぼしです。仕様として「1件でも未完了扱いなら通知しない」が意図なら現状で正しいです（**意図確認が必要**）。

**4. 具体的な対応案**
「実行中/待機中が0件になったら通知」に変えるのが自然です。
```csharp
var allFinished = DownloadQueue.All(j =>
    j.Status is DownloadStatus.Completed or DownloadStatus.Failed or DownloadStatus.Canceled);
var anyCompleted = DownloadQueue.Any(j => j.Status == DownloadStatus.Completed);
if (!allFinished || !anyCompleted) { _hasFlashedForAllCompleted = false; return; }
```

**5. 注意点**
通知の意味づけ（「全成功」なのか「全終了」なのか）が変わるため、UX 方針の確認が必要です。

## 【低】13. 設定画面のファイル名プレビューが拡張子 `.mp4` 固定

**3. 理由**
設定プレビューは「実挙動の確認用」なので、実際の拡張子と一致していないと誤解を招きます。

**4. 具体的な対応案**
`DefaultVideoFormat`（音声モード考慮が必要なら別途）を末尾に使う：`FilenamePreview = preview + "." + DefaultVideoFormat;`。`OnDefaultVideoFormatChanged` でも `UpdateFilenamePreview()` を呼ぶ。

**5. 注意点**
プレビュー専用の表示変更で、保存ロジックには影響なし。

## 【低】14. `GetThumbnailUrl` が最大解像度サムネを選び、一覧表示に過大画像を読む

**1. 改善すべき点**
[YtDlpClient.cs:264-295](YouTubeDownloader/Services/YtDlpClient.cs#L264) の `GetThumbnailUrl` は「最大 width」のサムネを選びます。プレイリストの各行や解析プレビューの小さなサムネ表示に対し、`maxresdefault`（1280×720 以上、数百KB〜MB）を毎回ダウンロードすることになり、行数が多いと帯域・メモリの無駄になります。

**2. 優先度**：低

**3. 理由**
表示は小サムネなので、過大解像度は読み込み遅延・メモリ増の割に画質メリットがありません。

**4. 具体的な対応案**
表示用は中庸サイズを選ぶ（例：width が一定値以下で最大のもの、または yt-dlp の `mqdefault`/`hqdefault` 相当を優先）。実装が重ければ現状維持でも可。

**5. 注意点**
WPF の `Image` 側で `DecodePixelWidth` を指定してデコード解像度を抑えるだけでもメモリは大きく削減できます（URL選定を変えなくてもよい）。

## 【低】15. `ValidatePaths`/`AutoDetectPaths` が UI スレッドで同期ディスク探索

**1. 改善すべき点**
[SettingsViewModel.cs:219-261](YouTubeDownloader/ViewModels/SettingsViewModel.cs#L219) の `ValidatePaths → AutoDetectPaths` は、`ExecutableLocator.FindExecutable`（PATH 全走査＋多数の共通パス＋アプリディレクトリの `File.Exists`）を UI スレッドで同期実行します。設定ロード時・パス参照ボタン押下時に走ります。

**2. 優先度**：低

**3. 理由**
頻度は低いものの、PATH にネットワークドライブや切断中ドライブが含まれると `File.Exists` がタイムアウトし、設定画面を開いた瞬間に UI が固まる可能性があります。

**4. 具体的な対応案**
自動検出を `Task.Run` でバックグラウンド実行し、結果プロパティだけ UI スレッドへ反映。検出結果は短時間キャッシュして連続呼び出しを抑制（前回 指摘14 のキャッシュ方針と整合）。

**5. 注意点**
非同期化すると検出表示にわずかな遅延が出ます。実害が小さいので「気づいたら直す」レベルで可。

## 【低】16. 24時間以上の再生時間表示がずれる（`TimeSpan.Hours`）

**1. 改善すべき点**
[VideoMetadata.cs:51-53](YouTubeDownloader/Models/VideoMetadata.cs#L51) の `DurationFormatted` は `ts.Hours` を使っています。`TimeSpan.Hours` は 24 時間で折り返すため、25 時間の動画が `1:00:00` のように表示されます。

**2. 優先度**：低（長時間配信などのエッジ）

**3. 理由**
表示専用とはいえ、長時間ライブのアーカイブ等で誤った時間が出ます。

**4. 具体的な対応案**
```csharp
var hours = (int)ts.TotalHours;
return hours > 0 ? $"{hours}:{ts.Minutes:D2}:{ts.Seconds:D2}" : $"{ts.Minutes}:{ts.Seconds:D2}";
```

**5. 注意点**
表示だけの修正です。長時間配信を対象外にする仕様なら優先度はさらに下がります。

## 【低】17. `ClearAll` 後も in-flight ジョブが完走するとライブラリへ保存され得る（エッジ）

**1. 改善すべき点**
[DownloadManager.cs:215](YouTubeDownloader/Services/DownloadManager.cs#L215) の `ClearAll` は `CancelAll()` 後に `_allJobs.Clear()` しますが、既に実行中の `ProcessJobAsync` はキャンセルが伝わる前にダウンロードを完走し得ます。その場合 `SaveVideoMetadataAsync` が走り、「クリアしたはずの動画」がライブラリ履歴に残ることがあります。

**2. 優先度**：低（タイミング依存のエッジ）

**3. 理由**
キャンセルは協調的（プロセス kill→OCE 伝播）なので、完了直前にクリアすると「クリアしたのに履歴に出る」軽微な不整合が起こり得ます。

**4. 具体的な対応案**
完了時の保存前に `cts.Token.IsCancellationRequested` を確認して保存をスキップする、もしくは現状仕様（「ダウンロードできたものは履歴に残す」）として許容する。**どちらを正とするか確認が必要** です。

**5. 注意点**
実害は小さく、無理に直すとロジックが複雑化します。現行が許容範囲なら据え置きで問題ありません。

## 【低】18. 参照共有された `VideoMetadata` をバックグラウンドスレッドで変更している

**1. 改善すべき点**
[UpdateLocalFilePath](YouTubeDownloader/Services/YtDlpClient.cs#L560) はダウンロードワーカースレッド上で `job.VideoMetadata.LocalFilePath` を書き換えます。この `VideoMetadata` インスタンスは UI にバインドされ（`DownloadJobViewModel`/`PlaylistItemViewModel`）、同じ参照が `SaveVideoMetadataAsync`（[DownloadManager.cs:115](YouTubeDownloader/Services/DownloadManager.cs#L115)）で永続化もされます。UI スレッドの読み取りと非同期の書き込みが無同期で同居しています。

**2. 優先度**：低

**3. 理由**
実害は出にくいものの、参照共有された可変モデルをスレッド跨ぎで変更する構造は将来の競合の温床です（`LocalFilePath` は文字列代入なので破壊は稀ですが保証はありません）。

**4. 具体的な対応案**
完了時の最終パスはイベント経由で UI スレッドに反映する、もしくは永続化用にコピーを作り、UI バインド対象と保存対象のインスタンスを分ける。最低限「完了時の最終パスだけ Dispatcher 経由で書く」程度でも十分です。

**5. 注意点**
大きな再設計は不要。`MaxConcurrency=2` での `SaveVideoMetadataAsync` 同時実行自体は `MetadataRepository` 側の mutex で保護済みであることは確認しています（=この指摘はインスタンス共有のみが対象）。
