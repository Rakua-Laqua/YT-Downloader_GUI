まず、添付ファイル2つをコンピュータsandboxにアップロードして内容を確認します。

実装案ファイルの続きを読みつつ、ZIPソースコードも展開して確認します。

実装案の検証のため、実際のソースコードを読み込んで両者の妥当性を確認します。

実装案の検証を進めるため、ダウンロードキューアイテムのXAML該当箇所と引数ビルダーの続きを確認します。

検証を完了させるため、ログサービスのインターフェースと残りのソース構造を確認します。

最後に、ダウンロード完了後にViewModelへ結果が伝搬される箇所を確認します。

ソースコード全体を読み終えました。実装案の内容と実際のコード構造を照合し、詳細に検証した結果をまとめます。

### AI1（概要型）の評価

**良い点:**

- `mp4` はコンテナ、`AV1` はコーデック、という区別を最初に明確にした設計思想が正確
- `container=mp4 / video=av1 / audio=aac` のように分けて保持する方針は概念的にクリーン
- テスト計画（単体テスト、正規化テスト、ffprobe JSONパース、ViewModelテスト、統合テスト）が実用的
- ソースファイルの行番号を正確に参照しており、実際のコードを読んでいることがわかる

**問題点:**

1. **コードが一切ない** — 方針と擬似ログしかなく、そのままでは実装に移れない
2. **`requested_formats` のパースが複雑** — yt-dlp の `requested_formats` はJSON配列/オブジェクトで出力されるため、パイプ区切りより解析が煩雑になる
3. **`MediaFormatProbeService` が過剰** — ffprobe を1回呼ぶだけの処理に独立クラスを作るのは過剰な抽象化。既存コードも `YtDlpDownloader` 内にffprobe関連ロジックを置くパターンに合致しない
4. **エッジケース未対応** — mp3/wav再エンコード時の音声コーデック不一致、マージ時のコンテナ変更、ffprobe未インストール時のフォールバック等が考慮されていない
5. **UIのツールチップ問題** — `StatusText` を `完了 !` に変更しつつ `StatusToolTip` を追加する案だが、既存XAMLでは `ToolTip="{Binding StatusText}"` となっているため、ツールチップにも `完了 !` が表示されてしまう。バインディングの両方を変える必要があることが明記されていない

---

### AI2（コード完備型）の評価

**良い点:**

1. **完全な実装コード** — 全ファイル分の変更が書かれており、そのまま実装に移せるレベル
2. **既存コード構造との整合性が高い** — `BuildDownloadArguments` の引数追加パターン、`ReadDownloadInfoFile` と同じパイプ区切り解析パターン、`ResolveFfmpegPath(settings)` の既存メソッド利用、`finally` ブロックでの一時ファイル削除パターン等、すべて既存コードの設計に沿っている
3. **コーデック正規化テーブル** — `avc1→h264`, `av01→av1`, `mp4a→aac` 等のマッピングが完備されており、yt-dlp と ffprobe の表記差を吸収できる
4. **ffprobe の呼び方が効率的** — `-select_streams v:0 -show_entries stream=codec_name -of csv=p=0` で必要最小限のデータだけ取得。JSON全体をパースする必要がない
5. **エッジケースの考慮表** — mp3/wav再エンコード、マージ時のコンテナ変更、ffprobe未検出、フォールバック再試行等を体系的に整理
6. **UI設計が優れている** — ⚠ を独立したTextBlockにし、`Visibility` と `ToolTip` を別々にバインドすることで、ステータステキストを汚さずにホバーで詳細メッセージを表示できる

**問題点・修正が必要な箇所:**

1. **⚠ 記号の二重表示バグ** — ViewModel の `StatusText` で `HasFormatMismatch ? "完了 ⚠" : "完了"` としつつ、XAML でも独立した ⚠ TextBlock を追加している。不一致時には `完了 ⚠ ⚠` と二重に表示されてしまう。**修正:** `StatusText` は `"完了"` のまま変更せず、XAML側の ⚠ TextBlock のみを使用する
2. **ffprobe の2回呼び出し** — 動画コーデックと音声コーデックで別々に `Process.Start` している。**修正:** 1回の `ffprobe -show_streams -of json` で両方取得するか、`-select_streams v:0 -show_entries stream=codec_name -of csv=p=0` と `a:0` を1回のプロセスで両方取るよう統合する
3. **`video:` フェーズの妥当性検証が必要** — `--print-to-file video:%(ext)s|%(vcodec)s|%(acodec)s` の `video:` フェーズは、yt-dlpがフォーマット選択後・ダウンロード後に発火する。`%(vcodec)s` / `%(acodec)s` は YouTube側が公開しているソースストリームのコーデックを返すため、ffprobe の結果と比較して「ソースと実ファイルのコーデックが一致しているか」を検証できる。ただし `%(ext)s` は**出力コンテナ**の拡張子（マージ後）を返すため、「ソースコンテナ vs 実ファイルコンテナ」の比較にはならない。コンテナ比較は実質的に常に一致する（ファイル名に `%(ext)s` が使われるため）。この点はドキュメント备注で明記すべき
4. **`TryDeleteTempFile` メソッドが存在しない** — AI2 は `TryDeleteTempFile(sourceFormatFile)` を呼んでいるが、既存コードにはこのメソッドがない。**修正:** 既存コードと同じインライン `try-catch` パターンで `File.Delete` する
5. **`SourceFormat` レコードの null 安全性** — `parts[0].Trim()` が空文字の場合、`null` ではなく空文字が返るが、`null` チェックが甘い。実害は低いが、`string.IsNullOrEmpty` チェックを追加すべき

### 総合判定

| 観点 | 推奨 | 理由 |
| --- | --- | --- |
| 実装の完成度 | AI2 | 全ファイルのコードが完備されており、既存コード構造との整合性が高い |
| 概念の正確さ | AI1 | コンテナ vs コーデックの区別を最初から明確にしている。ただし実装に反映されていない |
| エッジケース対応 | AI2 | mp3/wav再エンコード、マージ、ffprobe未検出などを考慮表で体系化 |
| テスト計画 | AI1 | 具体的なテストケースを提示。AI2にはテスト計画がない |
| UI設計 | AI2 | 独立した⚠ TextBlock＋専用ToolTipで、ステータステキストを汚さない |
| 既存コードとの統合 | AI2 | 引数追加パターン、パイプ区切り解析、finally削除等、既存のコーディング規約に忠実 |

### 結論：**AI2 をベースにすべき**

AI2 は実装可能なコードを完備しており、既存コード構造との整合性が高い。AI1 の概念的な長所（コンテナ vs コーデックの明確な区別、テスト計画）は AI2 に統合可能。

**採用すべき最終方針（AI2ベース + 修正）:**

1. AI2 のコードをベースに実装
2. **修正:** `StatusText` は `"完了"` のまま変更せず、XAML側の ⚠ TextBlock のみ使用（二重表示を防ぐ）
3. **修正:** ffprobe 呼び出しを1回に統合（`-show_streams` で動画・音声両方を取得）
4. **修正:** `TryDeleteTempFile` を既存のインライン `try-catch` パターンに置き換え
5. **追加:** AI1 のテスト計画（正規化テスト、ffprobe パースティング、ViewModel テスト、統合テスト）を取り入れる
6. **備考追加:** `%(ext)s` は出力コンテナ拡張子であり、ソースコンテナとは限らない点をコメントで明記。コンテナ比較は参考情報程度とする
