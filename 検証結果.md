いい着眼点です。「なぜリトライになるのか」を根本から潰せば初回成功率が上がります。まず引数生成ロジックを読んで、初回とフォールバックの差分を正確に把握します。

全体像が掴めました。核心はここです:

- **初回**: `--extractor-args youtube:lang=ja` のみ → **player_client 未指定**(yt-dlpのデフォルトclientに任せる)
- **再試行**: `player_client=tv,web_safari,android` を明示

つまり「デフォルトclientで失敗 → tv等を明示したら成功」というパターンでリトライが起きています。デフォルトclientの何が弱いのかを実測で検証します(年齢制限などの回復不能エラーで無駄にリトライしている可能性も併せて確認)。

まず、各clientの信頼性(PO Token要否・成否)を実測します。

決定的な差が出ました:

- **デフォルト(初回)**: `android_vr` + `web_safari` を使用。**web_safari は「SABR streaming強制でURL無しのためフォーマットがスキップされる」(yt-dlp issue #12482)** + PO Token要求。→ 使える formatが激減 → 初回失敗しやすい
- **`tv`**: PO Token不要・SABRスキップなし・クリーンに取得

`tv` が必要な mp4/avc1+m4a を実際に返すか、ニッチ動画のIP制限が解けたか、他clientとの比較を確認します。

`tv` は 144p〜**4K(2160p)まで avc1/av01/m4a 完備**で、フォーマット面の不足なし。一方ニッチ動画は `tv` でも bot-gate のまま = **client変更では回復不能エラーは直らない**ことも確認。最後に、フォーマット数の正確な比較と、`tv` での実ダウンロード(結合まで)を検証します。

`tv` での実ダウンロード(取得→結合→mp4出力)が**exit=0で成功**。フォーマット数も tv=27本(4Kまで完備)に対し **web_safari=7本(SABRで激減)**、android=1本(ほぼ死亡)。デフォルトが弱い原因が確定しました。

新リトライ用に検討中のclient文字列が有効か検証し、一時ファイルも片付けます。

すべて検証完了。多角的検証の結論が出ました:

| 検証 | 結果 |
|---|---|
| デフォルトclient(初回)の中身 | `android_vr`+`web_safari`。**web_safari は SABR強制でURL欠落 → 7本のみ**(#12482) |
| `tv` client | 27本・144p〜**4K完備**・PO Token不要・**実DL+結合 exit=0 で成功** |
| client変更でbot-gate回復? | **不可**(ニッチ動画は`tv`でもgate)→ 回復不能エラーのリトライは純粋な無駄&IP制限悪化 |
| 新リトライ候補 `default,tv_embedded,ios,mweb` | 有効(33本) |

**改善方針(2本立て):**
1. **初回から `tv` 先頭の安定client群を使う** → 「デフォルトが弱くて初回失敗→tvで再試行成功」を解消
2. **回復不能エラー(bot/年齢/非公開/429等)は再試行しない** → 無駄リトライ排除＋IP制限悪化を防止(分類器を活用)

実装します。まず分類器に「再試行で回復し得るか」フラグを追加(全面書き換え)。
